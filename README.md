# Демонстрационный учебный проект на Rust с GigaChat API

![Rust](https://img.shields.io/badge/rust-1.93.0-orange.svg) ![Rocket](https://img.shields.io/badge/rocket-0.5.1-blue.svg) ![GigaChat](https://img.shields.io/badge/gigachat-v1-green.svg)

Этот проект представляет собой минимально работающее веб-приложение на Rust, использующее фреймворк **Rocket** для создания локального HTTP-сервера и интегрированное с **GigaChat API** от Сбербанка. Проект разработан как учебное пособие для студентов, изучающих веб-разработку на Rust, и может служить основой для их собственных проектов.

## 🎯 Цели проекта

- **Показать лучшие практики**: продемонстрировать современные паттерны проектирования в Rust, такие как модульность, обработка ошибок и управление конфигурацией.
- **Обеспечить простоту освоения**: код написан максимально лаконично и подробно документирован, чтобы студенты могли легко понять его структуру и логику.
- **Создать гибкую основу**: проект можно легко адаптировать для других задач, изменяя конфигурацию и расширяя функциональность.
- **Продемонстрировать интеграцию с AI**: показать, как легко можно интегрировать современные AI-сервисы в веб-приложения на Rust.

## ✨ Основные возможности

- **Локальный HTTP-сервер**: на базе фреймворка Rocket.
- **Интеграция с GigaChat**: использует библиотеку `gigalib` для взаимодействия с GigaChat API.
- **Режим заглушки (Mock Mode)**: приложение может работать без доступа к GigaChat API, возвращая предопределенные ответы. Это полезно для разработки и тестирования.
- **Конфигурация**: все основные параметры (адрес сервера, настройки GigaChat, уровень логирования) вынесены в файл `config.toml`.
- **Документация**: код подробно документирован с использованием `rustdoc`, а также имеется эта документация в формате Markdown.
- **Модульная архитектура**: проект разделен на логические модули для лучшей читаемости и поддерживаемости.
- **Тесты**: включает интеграционные тесты для проверки работоспособности API.
- **Совместимость с учебным Web UI**: предусмотрено использование внешнего web-приложения, работающего с API на другом порту (требуется настройка CORS).

## 🚀 Быстрый старт

> 💡 **Для студентов:** Начните с [QUICKSTART.md](QUICKSTART.md) - там пошаговая инструкция на одной странице!

### 1. Требования

- **Rust**: убедитесь, что у вас установлен Rust (https://www.rust-lang.org/tools/install).
- **GigaChat API токен** (опционально): для работы с реальным GigaChat API получите токен на [developers.sber.ru](https://developers.sber.ru/).

### 2. Установка

```bash
# Клонируйте репозиторий
git clone <адрес_репозитория>
# Перейдите в каталог репозитория (имя может отличаться)
cd edu-rust-gigachat-app-backend
```

### 3. Конфигурация

```bash
# 1. Создайте файл .env из примера
cp .env.example .env

# 2. Откройте .env и вставьте свой токен GigaChat
# GIGACHAT_TOKEN=your_token_here
```

Вы также можете настроить приложение, отредактировав файл `config.toml`.

### 4. Запуск

```bash
# Запустите приложение
cargo run
```

Сервер будет запущен по адресу `http://localhost:8000`.

### 5. Тестирование

Вы можете протестировать API несколькими способами:

**Через `curl`:**

```bash
# Проверка здоровья
curl http://localhost:8000/health
# Ответ: {"status":"ok","version":"0.1.0","gigachat_enabled":true}

# Задать вопрос (английский для корректного отображения в консоли)
curl -X POST http://localhost:8000/ask \
  -H "Content-Type: application/json" \
  -d '{"question": "What is Rust?"}'
# Ответ: {"answer":"Rust is a systems programming language...","source":"mock ai service"}
```

**Через `HTTPie` (более удобный и наглядный):**

```bash
# Проверка здоровья
http GET localhost:8000/health

# Задать вопрос
http POST localhost:8000/ask question="What is Rust?"

# С явным указанием JSON
echo '{"question": "What is Rust?"}' | http POST localhost:8000/ask
```

**Установка HTTPie:**

```bash
# Windows (через Chocolatey)
choco install httpie

# macOS (через Homebrew)
brew install httpie

# Linux (Debian/Ubuntu)
sudo apt install httpie

# Linux (Fedora/RHEL)
sudo dnf install httpie

# Через pip (универсально)
pip install httpie
```

## Git и beads (учебный workflow)

В проекте используется встроенный issue‑трекер **beads**. Рекомендуемый режим:

```bash
bd init
bd hooks install
git config merge.beads.driver true
```

Задачи создаются с кратким описанием (`--description`), а синхронизация с git
обычно выполняется автоматически через хуки. При сбоях используйте `bd sync`.
`bd sync` **по умолчанию выполняет `git pull` и `git push`** — отдельный
`git pull` до/после него не нужен. `bd sync` коммитит **только**
`.beads/issues.jsonl` и **не**
коммитит изменения проекта. Поэтому изменения проекта нужно
закоммитить вручную после `git add -A`.

Требование к сообщению коммита (после `git add -A`):
- первая строка в формате `[<issue-id>] <type>(P#): <issue title>`;
- тело коммита совпадает с `--description`;
- после описания добавляется список изменённых файлов со статусами (A/M/D/R/C).

ID задач для этого репозитория имеют префикс `ergab-` (строчные буквы).

Памятка статусов (общепринятая git-нотация):
- A — файл создан (Added)
- M — файл изменён (Modified)
- D — файл удалён (Deleted)
- R — файл переименован (Renamed)
- C — файл скопирован (Copied)

Подробности: `lab/beads_guide.md`, `lab/git_version_control.md`,
`lab/git_github_setup.md`.

## ⚙️ Конфигурация

Основной конфигурационный файл - `config.toml`. В нем можно настроить:

- **`[server]`**: адрес, порт и окружение сервера.
- **`[gigachat]`**: включение/выключение GigaChat, модель, `max_tokens`, `temperature`.
- **`[logging]`**: уровень и формат логов.
- **`[application]`**: название, версия и описание приложения.

### Режим заглушки (Mock Mode)

Если вы хотите запустить приложение без доступа к GigaChat API, вы можете:

1. **Не устанавливать** переменную окружения `GIGACHAT_TOKEN`.
2. Установить `enabled = false` в секции `[gigachat]` файла `config.toml`.

В этом режиме приложение будет использовать `MockAiService` и возвращать предопределенные ответы.

## 🧩 Архитектурные решения

### Модульность

Приложение разделено на модули, каждый из которых отвечает за свою часть функциональности. Это упрощает навигацию по коду и его поддержку.

### Паттерн "Трейт-объект" для сервисов

Для работы с AI используется трейт `AiService`. Это позволяет легко переключаться между `GigaChatService` и `MockAiService` без изменения кода обработчиков. `AiServiceFactory` используется для создания нужной реализации в зависимости от конфигурации.

### Управление состоянием в Rocket

Конфигурация и AI сервис помещаются в управляемое состояние Rocket (`.manage()`). Это позволяет получить к ним доступ из любого обработчика.

### Обработка ошибок

Для обработки ошибок используется библиотека `thiserror`, а в Rocket настроены кастомные обработчики для ошибок 404, 422 и 500.

## ⚠️ Важные нюансы и подводные камни

### Работа с Rocket макросами

При использовании макросов Rocket (`#[get]`, `#[post]`, `#[catch]`) необходимо импортировать их явно:

```rust
use rocket::{get, post, catch};
```

Также для тестов потребуются макросы `routes!` и `catchers!`:

```rust
use rocket::{routes, catchers};
```

### Интеграция с gigalib

При работе с библиотекой `gigalib` важно учитывать следующие особенности:

1. **API методы**: Используйте `set_temp()` вместо `set_temperature()`
2. **Типы параметров**: `set_max_tokens()` принимает `u32`, а `set_basic_token()` принимает `&str`
3. **Проблемы с Send trait**: Клиент GigaChat не реализует `Send`, поэтому при работе в асинхронном контексте используйте `tokio::task::spawn_blocking`:

```rust
// Правильный подход для обхода проблем с Send
let result = tokio::task::spawn_blocking(move || {
    let runtime = tokio::runtime::Runtime::new().unwrap();
    runtime.block_on(async {
        // Ваш код работы с GigaChat
    })
}).await
```

4. **Архитектура сервиса**: Рекомендуется создавать клиент при каждом запросе, а не хранить его в структуре сервиса.

### Структура Cargo.toml

Обратите внимание на правильное размещение зависимостей:

- `async-trait` должен быть в секции `[dependencies]`, а не после других секций
- Избегайте дублирования секций `[dev-dependencies]`
- Порядок секций: `[dependencies]`, `[dev-dependencies]`, `[features]`, `[[bin]]`

### Тестирование

При написании интеграционных тестов:

- Rocket возвращает код `400 Bad Request` для невалидного JSON (не `422`)
- Импортируйте только используемые функции-обработчики и макросы
- Всегда используйте mock-сервис для изоляции тестов от внешних API

### Версия Rust

Проект требует Rust версии **1.93.0 или новее** из-за использования современных зависимостей. Если у вас установлена старая версия:

```bash
rustup update
```

### Предупреждения компилятора

Некоторые поля и методы специально оставлены для будущего расширения функциональности:
- `timeout_seconds`, `format`, `system_prompt` в конфигурации
- Методы `is_development()`, `is_gigachat_enabled()`

Эти предупреждения можно проигнорировать или удалить неиспользуемый код.
