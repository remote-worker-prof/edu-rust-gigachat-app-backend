# Руководство для студентов

Это руководство поможет вам понять, как работать с этим проектом, как его изменять и использовать в качестве основы для ваших собственных работ.

## 1. Первые шаги

Прежде чем начать, убедитесь, что вы выполнили все шаги из раздела [Быстрый старт](../README.md#🚀-Быстрый-старт) в основном файле `README.md`.

## 2. Как это работает?

Давайте разберем жизненный цикл одного запроса к нашему приложению.

1.  **Пользователь отправляет запрос**, например, `POST /ask` с JSON-телом.
2.  **Rocket** принимает запрос и находит соответствующий **обработчик** (`handler`) - в данном случае, функцию `ask` в `src/handlers/mod.rs`.
3.  Rocket автоматически **десериализует** JSON-тело в структуру `AskRequest` (из `src/models/mod.rs`).
4.  Обработчик `ask` получает доступ к **AI-сервису** через `&State<Box<dyn AiService>>`.
5.  Он вызывает метод `ai_service.ask(&request.question).await`.
6.  В зависимости от конфигурации, вызывается либо `GigaChatService`, либо `MockAiService`.
7.  Сервис выполняет свою работу (отправляет запрос к GigaChat или возвращает mock-ответ) и возвращает результат.
8.  Обработчик `ask` упаковывает результат в структуру `AskResponse` и **сериализует** ее в JSON.
9.  Rocket отправляет JSON-ответ пользователю.

## 3. Где и что изменять?

### Хочу добавить новый эндпоинт

1.  **Определите модели** (структуры запроса и ответа) в `src/models/mod.rs`.
2.  **Создайте новый обработчик** в `src/handlers/mod.rs`. Это будет `async` или синхронная функция, помеченная макросом Rocket (например, `#[get("/my-endpoint")]`).
3.  **Добавьте новый маршрут** в `main.rs` в секцию `.mount()`.

### Хочу изменить логику ответа AI

- Если вы хотите изменить **системный промпт** (инструкции для GigaChat), отредактируйте поле `system_prompt` в `config.toml`.
- Если вы хотите изменить **логику mock-ответов**, отредактируйте функцию `ask` в `MockAiService` в `src/services/mod.rs`.
- Если вы хотите добавить **новый AI-сервис**, создайте новую структуру, реализуйте для нее трейт `AiService` и добавьте логику ее создания в `AiServiceFactory`.

### Хочу добавить новую настройку

1.  **Добавьте поле** в одну из структур в `src/config/mod.rs`.
2.  **Добавьте соответствующую настройку** в `config.toml`.
3.  Теперь вы можете получить доступ к этой настройке через объект `AppConfig`.

## 4. Паттерны проектирования

### Модульность

Разделение кода на модули (`config`, `handlers`, `models`, `services`) помогает организовать код и делает его проще для понимания. Когда вы добавляете новую функциональность, подумайте, в какой модуль она должна попасть.

### Трейты для абстракции

Трейт `AiService` - это пример **абстракции**. Мы определяем *что* должен делать сервис (интерфейс), но не *как* он это делает (реализация). Это позволяет нам подменять реализации, что очень полезно для тестирования и гибкости.

### Внедрение зависимостей (Dependency Injection)

Вместо того чтобы создавать `AiService` внутри обработчика `ask`, мы получаем его извне (`&State`). Это делает обработчик независимым от конкретной реализации сервиса и упрощает его тестирование.

## 5. Советы по разработке

- **Начинайте с тестов**: Прежде чем реализовывать сложную логику, напишите тест, который проверяет, как она должна работать.
- **Используйте `cargo clippy`**: Это отличный инструмент для поиска распространенных ошибок и улучшения стиля кода.
- **Форматируйте код**: Используйте `cargo fmt` для автоматического форматирования кода в соответствии со стандартами Rust.
- **Читайте документацию**: Если вы не уверены, как работает функция или макрос, загляните в документацию (`cargo doc --open`).
- **Столкнулись с проблемой?**: Обязательно изучите документ [Часто встречающиеся проблемы](common_issues.md), где собраны решения типичных ошибок при работе с проектом.

## 6. Тестирование в режиме Mock

Приложение можно полноценно тестировать без настройки GigaChat API. Mock-сервис отвечает на вопросы по темам:

| Ключевые слова | Тема ответа |
|----------------|-------------|
| `rust` | Язык программирования Rust |
| `rocket` | Веб-фреймворк Rocket |
| `test` | Тестирование в Rust |
| `error` | Обработка ошибок |
| `api` | REST API |
| `json`, `serde` | Сериализация данных |
| `async` | Асинхронное программирование |
| `how...work` | Архитектура приложения |
| `hello`, `hi` | Приветствие и список тем |

**Примеры запросов через curl:**
```bash
curl -X POST http://localhost:8000/ask \
  -H "Content-Type: application/json" \
  -d '{"question": "What is Rust?"}'
```

**Примеры запросов через HTTPie (рекомендуется):**
```bash
# Простой синтаксис
http POST localhost:8000/ask question="What is Rust?"

# Проверка здоровья
http GET localhost:8000/health
```

**Установка HTTPie:** `choco install httpie` (Windows) или `brew install httpie` (macOS)

**Примеры запросов (PowerShell):**
```powershell
# Запрос к API
$body = '{"question": "What is Rust?"}'
Invoke-RestMethod -Uri "http://127.0.0.1:8000/ask" -Method POST -Body $body -ContentType "application/json"

# Или используйте готовый демо-скрипт
powershell -ExecutionPolicy Bypass -File demo_mock.ps1
```

## 7. Веб-интерфейс и CORS

При использовании отдельного web-интерфейса (например, проекта `edu-rust-gigachat-app-frontend`) UI и backend запускаются на разных портах (обычно `http://127.0.0.1:8080` и `http://127.0.0.1:8000`). Это разные origin, поэтому браузер блокирует запросы к API, если сервер не выдает CORS-заголовки и не обрабатывает preflight-запросы.

Минимальные требования к backend:
- Добавлять `Access-Control-Allow-Origin` (например, `http://127.0.0.1:8080`)
- Разрешить методы `GET, POST, OPTIONS`
- Разрешить заголовок `Content-Type`
- Обрабатывать preflight‑запросы `OPTIONS`

Последовательность CORS‑запроса в браузере:

1. UI отправляет `POST http://127.0.0.1:8000/ask`.
2. Браузер запускает preflight и делает `OPTIONS /ask`.
3. Backend отвечает `204 No Content` + `Access-Control-Allow-*`.
4. Браузер разрешает основной запрос.
5. Отправляется `POST /ask`, backend возвращает JSON + CORS‑заголовки.

Рекомендуемое решение для учебной среды описано в документе `docs/cors_for_webapp.md`. В нем приведен минимальный и воспроизводимый вариант настройки CORS для Rocket без сторонних крейтов.

## 8. SSH‑ключи и GitHub (пароль на каждом запуске терминала)

Если вы используете SSH‑ключ с паролем, то после перезапуска терминала пароль может
запрашиваться снова. Причина в том, что `ssh-agent` хранит ключи только в памяти процесса.
Чтобы пароль вводился один раз за сессию WSL, настройте «постоянный» агент по инструкции
в документе `lab/git_github_setup.md`.

## 9. Beads как issue‑трекер проекта

В проекте используется **beads** для хранения задач прямо в репозитории.
Рекомендуемый учебный workflow описан в документе `lab/beads_guide.md`.

**Best practice:** создавайте задачи с `--description`, чтобы сохранять контекст
и облегчать проверку выполненной работы.

Короткая памятка:

✅ Хорошо:
- `--description "Добавить эндпоинт /stats для отчёта преподавателю"`

❌ Плохо:
- `--description "Сделать задачу"`
