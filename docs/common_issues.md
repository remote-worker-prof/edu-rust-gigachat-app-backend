# Часто встречающиеся проблемы и их решения

> Этот документ содержит справочную информацию о типичных проблемах, с которыми могут столкнуться студенты при работе с проектом.

## 🔧 Проблемы при компиляции

### Ошибка: "feature `edition2024` is required"

**Причина:** Установлена устаревшая версия Rust.

**Решение:**
```bash
rustup update
rustc --version  # Должно быть 1.93.0 или новее
```

### Ошибка: "duplicate key in document root"

**Причина:** Дублирование секций в `Cargo.toml` (например, две секции `[dev-dependencies]`).

**Решение:** Убедитесь, что каждая секция встречается только один раз. Правильный порядок:
```toml
[package]
[dependencies]
[dev-dependencies]
[features]
[[bin]]
```

### Ошибка: "cannot find macro `routes` in this scope"

**Причина:** Не импортированы макросы Rocket.

**Решение:**
```rust
use rocket::{routes, catchers};  // Для конфигурации
use rocket::{get, post, catch};  // Для обработчиков
```

## 🌐 Проблемы с GigaChat API

### Ошибка: "Send` is not implemented"

**Причина:** Клиент GigaChat не потокобезопасен (не реализует `Send`).

**Решение:** Используйте подход с `spawn_blocking`:
```rust
let result = tokio::task::spawn_blocking(move || {
    let client = /* создание клиента */;
    let runtime = tokio::runtime::Runtime::new().unwrap();
    runtime.block_on(async {
        // Ваш асинхронный код
    })
}).await;
```

### Ошибка: "no method named `set_temperature`"

**Причина:** В библиотеке gigalib метод называется по-другому.

**Решение:** Используйте `set_temp()` вместо `set_temperature()`:
```rust
MessageConfigBuilder::new()
    .set_temp(0.7)  // ✅ Правильно
    // .set_temperature(0.7)  // ❌ Неправильно
```

### Ошибка типов при вызове `set_max_tokens` или `set_basic_token`

**Причина:** Неправильные типы параметров.

**Решение:**
```rust
.set_max_tokens(512)           // u32, не usize
.set_basic_token(&token)       // &str, не String
```

### Ответы не по теме (например, "Rocket" трактуется как ракета)

**Причина:** Вопрос сформулирован неоднозначно и может быть интерпретирован вне контекста Rust.

**Решение:** Явно указывать предметную область и назначение:
```text
Что такое Rocket как веб-фреймворк в экосистеме Rust? Кратко (3 пункта).
```

### Системный промпт не влияет на ответы

**Причина:** В используемой версии `gigalib` нет роли `system` (поддерживаются только `user` и `assistant`), поэтому «каноническая» передача системного промпта невозможна.

**Решение:** Применяется профессиональный компромисс: системные инструкции добавляются в начало пользовательского сообщения как скрытый префикс. Это позволяет влиять на стиль и контекст ответа, не раскрывая промпт пользователю.

### Прозрачность применения системного промпта в /ask

**Принцип:** Текст системного промпта не должен быть виден пользователю, но факт его применения должен быть прозрачно отражён.

**Решение:** В ответ `/ask` добавляется поле:
```json
{"system_prompt_applied": true}
```
Сам текст системного промпта не включается в JSON-ответ.

### GigaChat API недоступен (timeout / cannot reach)

**Причина:** сетевые ограничения или отсутствие доступа к API из текущей сети (firewall, proxy, VPN, блокировка порта 9443, региональные ограничения).

**Проверка доступа:**
```bash
curl -v https://ngw.devices.sberbank.ru:9443/api/v2/oauth
```

**Возможные решения:**
1. Проверьте прокси/фаервол и разрешите исходящие соединения на порт `9443`.
2. При необходимости используйте VPN с доступом к домену `ngw.devices.sberbank.ru`.
3. Если доступ недоступен, используйте mock‑режим:
   - удалите переменную `GIGACHAT_TOKEN` из окружения;
   - или установите `gigachat.enabled = false` в `config.toml`.

## 🧪 Проблемы с тестами

### Тест падает: assertion `left == right` failed (Status 400 vs 422)

**Причина:** Rocket возвращает 400 для невалидного JSON, а не 422.

**Решение:**
```rust
assert_eq!(response.status(), Status::BadRequest);  // 400
// не Status::UnprocessableEntity (422)
```

### Ошибка: "unused imports" в тестах

**Причина:** Неправильные или лишние импорты в тестовых модулях.

**Решение:** Импортируйте только то, что используете:
```rust
#[cfg(test)]
mod tests {
    use crate::handlers::{health, index};  // Только используемые
    use rocket::{routes, local::blocking::Client};
}
```

## 📝 Предупреждения компилятора

### Warning: "field is never read"

**Характер:** Не критично, поля оставлены для будущего расширения.

**Примеры:**
- `timeout_seconds` в `GigaChatConfig`
- `format` в `LoggingConfig`
- `system_prompt` в `ApplicationConfig`

**Варианты действий:**
1. Игнорировать (рекомендуется для учебного проекта)
2. Использовать поля в коде
3. Удалить неиспользуемые поля
4. Подавить предупреждение: `#[allow(dead_code)]`

### Warning: "method is never used"

**Характер:** Не критично, методы для удобства будущей разработки.

**Примеры:**
- `is_development()`
- `is_gigachat_enabled()`

**Решение:** Аналогично предыдущему случаю.

### Warning: "future incompatibility" (транзитивная зависимость)

**Причина:** Предупреждение связано с внешней библиотекой и её зависимостями (например, `nom v3.2.1` через `gigalib → tree_magic`). Это не ошибка проекта, а сигнал о потенциальной несовместимости в будущем.

**Решение для учебного проекта:**
1. Зафиксировать предупреждение как известное ограничение.
2. Объяснить, что исправление требует обновления внешних библиотек.
3. Не использовать форки/патчи без отдельного педагогического обоснования.

**Продвинутая опция (не рекомендуется для базового курса):**
- Использование `[patch]` в `Cargo.toml` или форка зависимости.  
Это усложняет воспроизводимость и поддержку проекта для студентов.

## 🧩 Проблемы с beads (issue‑трекер проекта)

### Ошибка: "LEGACY DATABASE DETECTED"

**Причина:** база beads была создана до версии 0.17.5 и не содержит идентификатор репозитория.

**Решение (для текущего репозитория):**
```bash
bd migrate --update-repo-id
```

**Если это свежий клон и база не нужна:**  
```bash
rm -rf .beads && bd init
```

### Коммиты и `bd sync` (важное правило)

`bd sync` коммитит **только** `.beads/issues.jsonl` и не включает изменения проекта.
Он **по умолчанию выполняет `git pull` и `git push`**, поэтому отдельный `git pull`
до/после него не нужен. Проектные правки нужно коммитить вручную после `git add -A`
с использованием формата сообщения `[<issue-id>] <type>(P#): <issue title>` и тела
из `--description`. Префикс ID задач — `ergab-`. Формальное описание процесса:
`agents-issue-workflow.cncf.yaml`.

## 🌐 Проблемы при работе с Web UI (CORS)

### Ошибка: "blocked by CORS policy" / "Failed to fetch"

**Причина:** Web-интерфейс и backend запущены на разных портах, что является разными origin. Браузер блокирует такие запросы без разрешающих CORS-заголовков и обработки preflight-запросов.

**Решение:** Включить CORS в backend Rocket и добавить обработку `OPTIONS`. Пошаговая инструкция приведена в документе `docs/cors_for_webapp.md`.

## 🚀 Проблемы с запуском

### Сервер не отвечает на запросы

**Возможные причины:**
1. Сервер не запустился - проверьте логи
2. Неправильный адрес/порт - проверьте `config.toml`
3. Firewall блокирует - добавьте исключение

**Проверка:**
```bash
# Убедитесь, что сервер запущен
cargo run

# В другом терминале - через curl
curl http://127.0.0.1:8000/health

# Через HTTPie (рекомендуется - более наглядный вывод)
http GET 127.0.0.1:8000/health

# Через PowerShell
Invoke-RestMethod -Uri "http://127.0.0.1:8000/health"
```

**Установка HTTPie (опционально):**
- Windows: `choco install httpie`
- macOS: `brew install httpie`
- Linux: `sudo apt install httpie` или `sudo dnf install httpie`

### Ошибка: "Address already in use"

**Причина:** Порт 8000 уже занят другим приложением.

**Решение:** Измените порт в `config.toml`:
```toml
[server]
port = 8001  # или другой свободный порт
```
